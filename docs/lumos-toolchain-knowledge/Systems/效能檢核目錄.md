# 效能檢核目錄——各平台 code review 階段該查的效能問題(2026-07 社群現況)

> **與 idioms skills 的分工**:idioms=完整壞例→好例(寫代碼時讀的教材);本目錄=審查/提醒層的**載重問**——idioms 裡標「⚠ 不可機檢」的條目優先入問(機器查不了才需要提醒),機械可查的歸 linter/analyzer。
> **定位**:與 [[Systems/linter精選目錄]] 同款「權威菜單」——code-loop 審查員鏡頭、impact 注入、pitfalls 追問的**內容源**,不是某專案的現況清單。每條標「機械可查」(可接確定性驗證器)或「人判提問」(進審查員/hook 提問)。
> **緣起**:2026-07-19 使用者要求搜「各平台該檢核的效能優化問題+如何在適當時機讓 Claude 自己感知該用最佳實踐」。

PRIOR-ART: 內容全數借官方/社群共識(Android 官方 Compose perf 文件、Microsoft ASP.NET Core Best Practices、Core Web Vitals 2026 checklist);時機點設計借 Vercel 56% 跳過實證+Claude Code hooks 保證注入模式=lumos impact hook 既有哲學;裁定=borrow(收錄共識,不自創檢核項)。

## Android / Jetpack Compose(消費端:Citrus_KDS、mOrangePos)

| 檢核項 | 型態 | 說明 |
|---|---|---|
| 不穩定型別導致重組失控 | **機械可查** | compiler stability report(compose-metrics-adapter 軌道);unstable=Compose 放棄 skip、每次全重畫 |
| 昂貴計算有沒有 `remember` | 人判提問 | 熱 composable 內的計算/排序/過濾 |
| lazy 列表有沒有穩定 `key` | 人判提問 | 無 key=插刪時整段重組 |
| 高頻狀態有沒有 `derivedStateOf` | 人判提問 | scroll offset 這類快速變化狀態直讀=每幀重組 |
| 「讀過的狀態又寫」 | 人判提問 | 重組死循環模式 |
| state read 能否延遲(lambda 包裝) | 人判提問 | 把讀往下推,縮小重組範圍 |
| app 專屬 Baseline Profile+R8 | 人判(release 檢核) | 預設 profile 不夠;launch/捲動關鍵路徑該收錄 |
| **互不依賴的等待必須並行** | 人判提問(kotlin-idioms R1 頭號,不可機檢) | 串行 await 互不依賴的 suspend=白等;coroutineScope+async 並行 |
| **協程 scope 洩漏/誤用** | 人判提問+lint 可查部分 | GlobalScope 裸用(lint 有規則)/自建 scope 沒跟生命週期(該用 viewModelScope/lifecycleScope)/離開畫面沒取消=洩漏+背景浪費 |
| Dispatchers 誤用 | 人判提問 | IO 密集用 Default(佔運算池)/運算密集用 IO(執行緒爆量)/main 上做重活 |
| 結構化並發與取消傳播 | 人判提問 | async 沒 await 吞例外/withContext 内 catch CancellationException 吞掉取消/長迴圈沒 ensureActive |
| **大集合鏈式操作語法糖** | 人判提問 | 長鏈 map/filter 每步生中間 List→大集合該 asSequence;字串拼接用 buildString;重複 list+list 用 buildList |
| **記憶體洩漏(Context 持有)** | 人判提問+工具可抓 | 2026 仍是頭號隱形殺手:單例/SDK/companion 持 Activity Context 或 View→畫面銷毀 GC 收不掉(該給 applicationContext);工具=LeakCanary+StrictMode detectAll() |
| **value class 裝箱三觸發** | 人判提問 | @JvmInline 零成本被泛型/可空/多型觸發裝箱吃掉(另 reflection/序列化);原始型別陣列 vs 裝箱版實測 23x 差 |
| primitive array/forEach on range | **機械可查(detekt Performance 規則集)** | ArrayPrimitive/ForEachOnRange 現成規則——歸 linter 不進提問層(分工規矩) |
| **shareIn 放函式回傳=重複熱流洩漏** | 人判提問(kotlin-idioms R5 已禁) | 每呼叫建新熱流,多畫面各叫一次漏一堆;R5「shareIn/stateIn 禁止放函式回傳」的洩漏面注解 |
| **Channel/Flow 適用性** | 人判提問 | 熱冷選型:狀態=StateFlow/事件廣播=SharedFlow/單一消費者佇列=Channel/按需資料=cold Flow——選反=漏事件或黏著重放;背壓 buffer/conflate/collectLatest;flowOn 放錯層無效;UI 收集要 repeatOnLifecycle 否則背景繼續收 |

## .NET / ASP.NET Core 後端(消費端:LandmarkMember)

| 檢核項 | 型態 | 說明 |
|---|---|---|
| 熱路徑阻塞異步(`.Result`/`.Wait()`) | **機械可查**(analyzer:VSTHRD/regex) | csharp-idioms 已有條目;審查時機由本目錄接 |
| CancellationToken 斷鏈 | **機械可查**(analyzer) | 同上 |
| N+1 / 查詢數失控 | **機械可查**(S2 查詢數斷言) | 「操作 X ≤ N 條查詢」寫成測試,[[Projects/code階段強化_計劃]] S2 |
| 只撈需要的欄位(投影) | 人判提問 | Dapper `SELECT *` 進大表/EF 未投影 |
| 熱讀路徑該不該上 HybridCache | 人判提問 | .NET 9;API 回應/昂貴查詢,18x 實測案例 |
| 熱路徑短命大物件/該用 Span/ValueTask | 人判提問 | GC 壓力;dotnet-counters 量測佐證 |
| **互不依賴的等待必須並行** | 人判提問(csharp-idioms R1 頭號,不可機檢) | 串行 await 互不依賴的 Task=白等;Task.WhenAll |
| **大結果集分頁/下推** | 人判提問(csharp-idioms R11) | 過濾聚合下推到資料層,別撈全表進記憶體處理 |
| **LINQ 重複枚舉/中間集合** | 人判提問+analyzer 可查部分 | IEnumerable 被多次 foreach=查詢重跑(該 ToList 一次);長鏈每步生中間集合;熱路徑 LINQ vs 迴圈取捨 |

## Vue / 前端(消費端:Vue 專案)

| 檢核項 | 型態 | 說明 |
|---|---|---|
| INP ≤200ms:互動會不會卡主線程 | 人判提問+lab 量測 | 2026 最常掛的 vital(43% 不及格);長任務拆分/防抖 |
| LCP 元素誤加 lazy loading | 人判提問 | 自殺式優化:對要優化的目標 defer 請求 |
| 新依賴 bundle 重量/能否動態載入 | **機械可查**(bundle 分析) | code splitting/dynamic import |
| 圖片寬高缺失(CLS) | **機械可查**(lint 規則) | lazy 圖也要寬高保留空間 |
| computed vs watch 誤用 | 人判提問 | vue-idioms 已有條目 |
| **互不依賴的等待必須並行** | 人判提問(vue-idioms R1 頭號,不可機檢) | 串行 await 互不依賴的請求=白等;Promise.all |
| **async watch 競態** | 人判提問(vue-idioms R3,最隱蔽) | 快速切換舊回應晚到蓋掉新資料;AbortController+onCleanup |
| **watch 成本/v-for per-item prop** | 人判提問(vue-idioms R8/R11) | 深層 watch 大物件要有意識;v-for 子項 prop 預算 per-item 值;千項級虛擬化 |
| LCP≤2.5s / CLS≤0.1 | lab/監控 | 門檻值 2026 不變 |

## 時機點設計(讓 Claude「自己感知」的機制,非靠自覺)

**實證錨**:Vercel 量測——靠模型自己「想起要查規範」,**56% 情況會跳過**;hook 保證注入與模型意願無關。同 lumos「機械把關>靠自覺」與 impact hook 既有哲學。**提問式>陳述式**(pitfalls 逼答哲學):在對的時機丟「問題」比丟「規則全文」更能觸發判斷。

| 時機 | 機制 | 內容 |
|---|---|---|
| ①動手改代碼前 | impact hook(既有)按棧擴充 | diff 碰 `.kt`→注入 Compose 三問;`.cs`→async/N+1 三問;`.vue`→INP/bundle 三問 |
| ②終審 code-loop | 檢核表當審查員鏡頭(仿 test-layers 鏡頭模式) | 本目錄該棧段落餵 reviewer;機械可查項走確定性驗證器三通道 |
| ③蓋章前 | pass 留痕紀律 | 命中效能類的改動,pass --note 須含對應檢核問題的答案(同接受理由紀律) |

落地計劃:[[Projects/pitfalls棧別效能追問_計劃]]。

## 誠實邊界
- 2026-07 快照;門檻值(INP/LCP)與工具生態會變,半年宜重搜(同 linter精選目錄慣例)。
- 「人判提問」項無機械 oracle——提問保證被看見,不保證答對;答案品質靠審查員+辯方。
- 本目錄不含 prod 監控層(歸消費端 ops);code 階段止於 CI 可跑+審查可問。
