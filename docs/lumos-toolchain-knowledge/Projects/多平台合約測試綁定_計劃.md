# 多平台合約測試綁定_計劃

> 讓**單一知識圖譜**能把 ★INVARIANT★ 的 `[test:]` 綁到**不同平台**的測試（後端 C#、Android Kotlin、Android Maestro E2E、網頁 Playwright），而不是現在「一個 repo 只能一個 profile、只掃自己 repo」。

## 問題（為什麼非做不可）

KDS 圖譜（`Citrus_KDS/docs/kds-knowledge/`）記錄的是**橫跨前後端的單一系統**（例：invariant「震動器號碼來自**後端** OrdersList 的 OrderNote」，測在**前端** mapping）。但現行 Check T 綁定機制有三道牆：

1. **一個 repo 只能一個 `test_profile`**：KDS config = `kotlin-junit`，Check T 只認 Kotlin `@Test`；後端 C# 合約想綁 `[Fact]` 會被判 dangling/fake。
2. **`discover_test_methods` 只掃 `repo_root`**（= 圖譜所在 repo）：後端 C# 測試在 `Compass_KDS`（另一 repo），物理上掃不到。
3. **兩 repo 都無 `.lumos/guard-templates/`**：`guard scaffold` 對誰都跑不起來（`bind` 現成測試可，`scaffold` 產 stub 不行）。

延伸缺口：**`test_profile` 機制本身（P5，csharp-xunit/kotlin-junit）在 toolchain 圖譜裡沒有 Systems 節點**——本計劃順帶補建。網頁前端**目前無 repo 存在**（找不到 `playwright.config.*`）→ Playwright profile「先內建、KDS 暫不接線」。

## 設計（方向已與人確認：多根＋多 profile，單圖譜跨平台，全平台）

### D1. config.json 從單 profile → `platforms` 多根多 profile map

```json
// Citrus_KDS/.lumos/config.json
{ "default_platform": "android",              // 無前綴 [test:X] 的歸屬平台(見 D2)
  "platforms": {
    "android":  {"profile": "kotlin-junit", "root": "."},
    "maestro":  {"profile": "maestro",      "root": "."},
    "backend":  {"profile": "csharp-xunit", "root": "../Compass_KDS"}
    // "web":   {"profile": "playwright",   "root": "../<web repo 待確認>"}
} }
```

- **`default_platform` 解析規則**（審計 r1 指出原 schema 漏定義）：
  - 明確給 → 用它；未給且 `platforms` 只有 1 個 → 即該平台；未給且 `platforms` 有多個 → **載入時報錯**（要求補 `default_platform`，不猜、不靜默 fallback，避免現有裸 ref 誤綁）。
  - `default_platform` 指向不存在的 platform 鍵 → 載入報錯。

- **向後相容鐵則**：舊 `{"test_profile":"kotlin-junit"}` 與無 config 必須照舊運作（現有唯一 invariant `orderNoteKeyMapsToBuzzerNo` 不能斷）。`platforms` 缺席 → 退回單 profile 舊路徑。
- `root` 相對圖譜 repo 解析，允許 `../`；解析式明寫為 **`(repo_root / root_str).resolve()`**（審計 r4：非 `Path(root_str).resolve()`——後者相對 cwd，cwd≠repo_root 時結果錯）+ 存在性檢查（不存在 → 該平台 discover 跳過並**大聲警告**，不靜默）。

### D2. `[test:]` 標記加平台前綴 `[test:<platform>:<name>]`

```
KEY:★INVARIANT★ ... [test:android:orderNoteMapsToBuzzer]
KEY:★INVARIANT★ ... [test:maestro:checkout_happy_path]
KEY:★INVARIANT★ ... [test:backend:AdditionSecondLangQty]
```

- **解析演算法（每條 ref 獨立判定）**：`TEST_REF_RE` 抓 `[test:...]` 括號內全文 → 現行按逗號 split 成多個 ref → **對每個 ref 再看有無 `平台:` 前綴**（第一個 `:` 前是已定義的 platform 鍵才算前綴，否則整串當測試名）。
  - `[test:android:X,backend:Y]` → `{android: X}`, `{backend: Y}`。
  - `[test:android:X,Y]` → `{android: X}`, `{default_platform: Y}`（逐段獨立，無前綴段落回 default_platform）。
  - `[test:X]` → `{default_platform: X}`。現有裸 `[test:orderNoteKeyMapsToBuzzerNo]` 即走此路，仍有效。
  - ⚠ 冒號歧義：C#/Kotlin 測試名不含 `:`，故「第一段是否為 platform 前綴」以**是否為已定義 platform 鍵**判定，非只看有無 `:`。
- **向後相容（單一規則，D2/T2/T3 一致）**：**前綴解析僅在 config 有 `platforms` 鍵時啟用**。`platforms` 缺席（舊 `{"test_profile":...}` 或無 config）→ `resolve_test_refs` **不做冒號前綴切分**，整串（含任何冒號，雖 C#/Kotlin 名不含）即方法名，對單一 method set 比對；`[test:foo:bar]` 在 legacy 下 = 方法名 `foo:bar` → dangling（**不報錯**）。多平台模式下才有「像前綴但非已定義 platform 鍵 → 報錯」語意。
- **⚠ legacy 信號表示法（審計 r4 + opus 辯方拍板）**：`resolve_test_refs` 是否切分**以「config 原本有無 `platforms` 鍵」為信號**，不以傳入 dict 是否為空推斷。落地取辯方推薦解：legacy 時**傳空 dict `{}` 給 `resolve_test_refs`**（→ 不切分）；`load_platforms` 在 legacy 合成的「單一條目 dict」**只供 discover/profile 選擇**用，不當切分信號（否則「真的只 1 平台的多平台 config」會與 legacy 混淆）。D2/T2/T3 一律照此表示，消除三節對 legacy `platforms` 值（空 vs 單一條目）的描述分歧。
- Check T 對每條 ref：依判定的 platform 選 `{profile, root}` → 只在該平台的 method set 裡找。找不到 → dangling；在 code 但非測試方法 → fake（同現行語意，範圍限該平台）。
- 前綴是「像前綴但非已定義 platform 鍵」（如 `[test:foo:X]` 而 `foo` 不在 `platforms`）→ **明確報錯**（不當測試名、不 fallback，避免誤綁）。
- **API 設計（審計 r2 指出級聯風險，明定以免破壞呼叫點）**：現行 `invariant_test_refs(inv_text)` 回**字串 list**（`["orderNote..."]`），多處依賴（`strip_test_refs:1064`、`cmd_guard_bind` 的 `method in invariant_test_refs(...)` 檢查、`cmd_guard_list` 的 `[test:{','.join(refs)}]` 重建 tag、`cmd_guard_trace` 的 `re.escape(mth)`）。**不改其簽章/回傳型別**（避免級聯破壞），改為**新增** `resolve_test_refs(inv_text, platforms, default_platform)` → 回 `[(platform, name)]` tuple list，只給需要平台解析的新邏輯用（Check T、classify_invariants、archive）。`invariant_test_refs` 維持原樣供 tag 重建/顯示用。`platforms` 為空（單 profile 舊路徑）時 `resolve_test_refs` 回 `[(default_or_single, name)]`，向後相容。

### D3. 新增內建 profile：`maestro`（Android E2E）

Maestro 測試單位 = YAML flow 檔；天然身份是**檔名**，`method_re` 從不看檔名 → 錯配。**採「綁 `name:` 欄位」路線**（沿用現有 content-regex discover 的**比對邏輯**，不新增第二套掃描機制；但 discover 主迴圈需加一個 per-file 過濾，見下 `file_must_match`——**不是零程式改動**）：

```yaml
appId: com.example.myapp
name: checkout_happy_path      # ← 綁定目標；慣例：必寫、識別字安全（無空白）
tags: [checkout]
---
- launchApp
```

- profile（**完整 dict，補齊 r2 指出的漏項；對照 `kotlin-junit` `scripts/lumos:980`**）：
  ```python
  "maestro": {
      "exts": {".yaml", ".yml"},
      "method_re": re.compile(r"(?m)^name:\s*[\"']?([A-Za-z_]\w*)[\"']?\s*$"),  # 行尾錨,見下
      "scaffold_ext": ".yaml", "dir_mode": "rglob", "rglob_under": None,
      "file_must_match": re.compile(r"(?m)^appId:"),   # 見下
      "attr_hint": "Maestro flow name:", "fail_hint": "assertVisible(不存在的哨兵)",
      "dirs": {"pure": ([], []), "behavioral": ([".maestro"], []), "state": ([".maestro"], [])},
  }
  ```
- **新增 discover knob `file_must_match`（選填，`re.Pattern` 或 None）**：只認內容**行首**含 marker 的檔（Maestro = `(?m)^appId:`），否則 Android repo 一堆 `.yml`（GitHub Actions 的 `name: Build and Deploy`）會被誤收成測試名。**用錨行首 regex 而非純字串 `in`**（審計 r2：純 `"appId:" in content` 會被歷史註解 `# appId: ...` 誤觸）。實作 = `discover_test_methods` 主迴圈讀檔**後、去 C 註解之前**插一次 `file_must_match.search(txt)`（審計 r4：明定插在去註解前，對 `^appId:` 兩處結果同、但釘死免未來 profile 依賴隱含順序）；不符即 `continue` 跳該檔。**現有 `csharp-xunit`/`kotlin-junit` profile dict 無此欄位** → 實作用 `profile.get("file_must_match")`（審計 r4：直接 `profile["file_must_match"]` 會 KeyError），None 即不過濾、行為不變、向後相容。
- ⚠ **行尾錨 `\s*$` 消假陽性（審計 r3）**：原 `["']?` 為選填閉合引號，`name: should checkout flow` 會**部分捕獲** `should`（比 Playwright 更寬鬆、製造假陽性 `[test:maestro:should]` 被誤判 real）。加 `\s*$` 要求 title 後即行尾 → 多字 name **NO MATCH**（與 D4 Playwright 對稱），`name: checkout_flow`（含/不含引號）仍正常 MATCH。多字 `name:` 不可綁定即設計意圖。
- ⚠ **已知 trade-off（審計 r2）**：`discover_test_methods` 現行去 C-style 註解（`//…`、`/* */`，`scripts/lumos:1206`）對 YAML 不完美——YAML 值域的 `//`（如 `url: https://…`）會被誤刪。Maestro `name:` 行不含 URL 故不影響擷取，但這是「discover 對 YAML 非 100% 乾淨」的已知限制，非隱性正確；v1 接受（`name:` 擷取安全），未來要掃 YAML 其他欄位再議是否 profile-aware 去註解。
- ⚠ **釐清 `dirs` vs discover 範圍**（審計 r1 指出的易誤解點）：`profile["dirs"]` 只被 `_detect_test_dir` 讀（scaffold 輸出目錄偵測），**`discover_test_methods` 不讀 `dirs`**——它 `os.walk` 全 repo、只靠 `CODE_SKIP_DIRS`＋副檔名＋（本次新增的）`file_must_match` 過濾。所以限縮 Maestro 掃描範圍**只能靠 `file_must_match`**，`dirs.behavioral=[".maestro"]` 對 discovery 無效（僅影響 scaffold 落點）。
- 慣例代價：guard-bound flow 的 `name:` 必須是識別字（配合 `IDENT_RE`／`guard scaffold --method` 限制，同 Kotlin 反引號名的既有限制）。

### D4. 新增內建 profile：`playwright`（網頁 E2E，先內建不接線）

- Playwright 測試：`.spec.ts`/`.test.ts`，`test('title', async ({page})=>{})` / `test.describe(...)`；識別子 = title 字串（自由文字、可含空白）。
- profile：`exts:{.ts,.tsx,.js,.jsx}`、`scaffold_ext:.spec.ts`、`method_re = \btest(?:\.describe)?\(\s*["']([A-Za-z_]\w*)["']`（抓 `test(`/`test.describe(` 第一參數，**只認識別字安全的 title**——見下）。
- **⚠ title 自由文字 vs `IDENT_RE` 的結構性死結，及其乾淨解（審計 r1 提出、r2 追問矛盾）**：Playwright/Maestro 識別子天生可含空白（如 `test('should show menu')`），但 `guard scaffold/bind` 用 `IDENT_RE=^[A-Za-z_]\w*$`、且 `[test:]` 綁定名也需識別字。本計劃**不改 `IDENT_RE`**（牽動 C#/Kotlin 既有守衛），取一致的明確界線：
  - **guard-bound 的 E2E 測試，其 title/`name:` 必須是識別字安全 token（無空白）**——寫進 docs 的可測性慣例。`method_re` 的 capture group `([A-Za-z_]\w*)` 據此設計：只擷取識別字 title。
  - **含空白 title（如 `'should show menu'`）= 明確不支援 guard 綁定**（與 Kotlin backtick 名同限制）。此時 `method_re` 對它**完全不匹配（NO MATCH）**（capture group `([A-Za-z_]\w*)` 吃完 `should` 後需緊接閉合引號卻遇空白，回溯也接不上 → 整體不匹配；經 opus 辯方 python 實測確認，非「抓到 should」）→ 不會污染 method set、也對不起任何手寫 `[test:]` → **設計上就不該綁**。要綁就把該測試 title 改成識別字。此 NO MATCH **恰是設計要的行為**（多字 title 不可綁），非 bug。
- **無 web repo → 本輪只定義 profile + 單元測試 fixture，不進任何 config**；待 web repo 出現再接線。
- **撰寫 workflow 註記（Playwright MCP，與 D5 Maestro MCP 對稱）**：官方 `@playwright/mcp`（微軟）可讓 Claude Code 用瀏覽器 accessibility snapshot 探索網站、產 `.spec.ts`、跑綠——即「把紅燈 stub 填到綠」的工具。定位同 Maestro MCP：**開發/撰寫期工具、非合約守門**（Check T／CI 才守門）。接 web repo 時採用。（此註記為收斂後 additive，與 D5 已審過的 Maestro MCP 註記同款、非規範性。）

### D5. guard-templates（scaffold 紅燈 stub）

- 各平台一份 `.lumos/guard-templates/<type>.tmpl`（技術棧專屬，lumos 不內建）。
- Maestro `behavioral.tmpl`：紅燈哨兵 = 一個必失敗的 `assertVisible`（對應 C# `Assert.Fail`/Kotlin `fail()`）。
- 本計劃提供**範例 template**（放 toolchain repo 供參考/測試），實際落到 `Citrus_KDS`/`Compass_KDS` 各自的 `.lumos/guard-templates/`。
- **撰寫 workflow 註記（Maestro MCP）**：`guard scaffold` 產出的是紅燈 flow stub（哨兵必失敗 `assertVisible`），把它**填到綠**建議用官方 [Maestro MCP server](https://docs.maestro.dev/get-started/maestro-mcp)——Claude Code 經 MCP 觀察真機/模擬器的 live view hierarchy＋截圖、產正確 `tapOn`/`assertVisible` flow YAML、當場在裝置上跑驗。定位：**開發/撰寫期工具，非合約守門**——它補的是「真的跑起來（Check T 天花板①的另一半）」，不改變 lumos 綁定存在性檢查與 CI gate 的角色。前置：要模擬器/真機（同天花板②「無裝置才 skip」）。

## 天花板（誠實鐵則，寫清楚免被誤讀）

1. Check T 只驗「測試識別子存在且認得到」，**不驗它會綠**——同現行驗 C#/Kotlin 方法存在性。真正跑綠是 CI 的事（guard 表 behavioral/state → lab deploy）。綁定可行性 ≠ CI 執行。
2. Maestro/Playwright E2E **要裝置/瀏覽器才跑** = 現有 `SkippableFact`「無 DB 才 skip」的類比（「無裝置才 skip」），「不可靜默跳過」鐵則照樣適用——屬 CI 編排，不在本計劃「綁定」範圍。
3. 多根解析允許 `../`，但**只讀不寫**跨 repo（discover 是唯讀掃描）；scaffold 寫入仍落在明確 `--out` 或該平台 root 下偵測到的測試目錄。
4. **Check K × 多平台 comma ref 交互（審計 r2）**：Check K（`scripts/lumos:716`）計 `TEST_REF_RE.findall(inv)` 的**標記個數**（防逗號繞過），故 `[test:android:X,backend:Y]` 仍算 **1 個標記** → 標了 `★COMBO★` 的跨平台雙綁仍會觸發「只綁 1 個 [test:] → 補組合」軟提醒。這是**已知且可接受**的交互（COMBO 要的是「多情境」，跨平台雙綁是否算多情境屬語意判斷，軟提醒不擋、不誤傷）；本計劃**不改 Check K**，僅在此註明免 CI 使用者困惑。
