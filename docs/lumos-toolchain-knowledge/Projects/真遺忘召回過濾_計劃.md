# 真遺忘召回過濾_計劃

> **狀態**：設計中（r1 light 審計已修，範圍縮為 search-only），待重過 design-loop → TDD。緣起：2026-07-24 治理調研（GateMem, arXiv 2606.18829）照出 lumos「supersede 只蓋章、召回時照樣遞給 agent」——標記 ≠ 真遺忘。

## 白話問題

lumos 是「這套系統的筆記本」，AI 每次接手前第一件事就翻筆記找相關頁。一條規則作廢時只在那頁蓋「作廢」章，**但那頁還留在筆記本、翻筆記時照樣被遞出來**——AI 可能讀到一條已死掉的規則卻當它還活著。真遺忘＝翻筆記時**預設別把作廢頁遞出去，但想查還查得到、且被告知「藏了幾筆」**。

## 洞的形狀（讀碼確認）

| 召回路徑 | 現況 | 洞？ | 本計劃 |
|---|---|---|---|
| `search` | 只**降權**不排除，`_RANK_STATUS_MULT` 凍在 `1.0`＝關 | ✗ 漏 | **← 第一刀，只做這條** |
| `context` 推薦（`_reco_fused`） | 無 status 過濾，且走 BFS 多跳 | ✗ 漏 | **緩**（先決定遍歷語意，見範圍刀②） |
| `impact` 注入 | 無 status 過濾 | ✗ 漏（但**故意不補**） | **不碰**（作廢＝事故記憶，見範圍刀①） |
| `doctor` Check S | 已 `skip`（但 stale+superseded 綁一起） | —（**非召回路徑**，是稽核義務豁免） | 只借它 `if status in(...):continue` 的碼型、不抄綁定（F4） |

## A方案第一刀：只做 search 的「預設排除、明示可召回」

`cmd_search` 產候選集時，預設濾掉 `status: superseded` 的節點；加 `--include-superseded` 旗標可顯示。**藏掉時印一行**「（已隱藏 N 筆作廢結果，`--include-superseded` 顯示）」——遺忘變成**看得見、可審計**（滿足 GateMem「真遺忘須可驗」）。

### 濾網插點＝「命中確認後、三路分岔前」（r2 F2＋std F6 精修）
`cmd_search` 有**三條輸出路徑**：預設 `ranked`／`--legacy`（字母序）／`--regex`（dispatch 層強制關 ranked）。濾網**必須插在「確認該節點命中（過了 `if not hits: continue`）之後、三路分岔之前」**那一點——不是泛稱「共用迴圈」（std F6）：
- 只插 `ranked` 分支 → `--legacy`／`--regex` 漏出作廢節點（洞沒補到卻看似補了）。
- 插在命中確認**之前**（迴圈頂）→ 隱藏數會變成「全庫 superseded 總數」而非「本次查到、被藏的筆數」（測試 6 擋不住此錯算）。
- ∴ **`hidden_superseded` ＝「命中 ∧ superseded ∧ 未帶 --include-superseded」被濾掉的結果筆數**，不是掃過的作廢節點數。

### 隱藏提示的輸出通道（r2 F3＋std F4/F5 精修）
- **提示一律走 `stderr`**（不污染 stdout 的機器可讀輸出）——**所有模式皆印，含 `--files-only`**（stderr 本就不會混進 stdout 的檔名管線，無理由取消；取消＝對 files-only 重開靜默遺忘，std F4）。
- **`--json`**：在結果物件加 `hidden_superseded: N` 欄位（stdout 仍是合法 JSON）＋ stderr 提示照印。
- **`--legacy`／`--regex`（非 ranked）**：`--json` 目前只在 ranked 分支生效（std F5）——本刀**明定**：非 ranked 模式的 `hidden_superseded` 只走 stderr 提示（不硬塞半套 JSON）；`--regex --json` 這種「json 被忽略」的既有行為不在本刀擴充範圍，維持現狀＋stderr 提示兜底可審計性。

### 逃生口＝明示旗標，不是「點名放行」（r1 審計 F2）
原設計寫「query 直接點名該節點時仍放行」——**search 沒有這個機制**：`cmd_search` 只拿 query 比對節點**內文**，從不比對節點名/路徑（無 by-id 概念）。所以 search 的召回口**只能靠 `--include-superseded` 旗標**。（context/impact 才有「傳節點 id」的天然點名，但那兩條本刀不做。）

### ⚠ 樞紐陷阱：stale ≠ superseded（整個方案成敗在此）
- `superseded`（合約被取代）→ **該遺忘**、預設排除 ✓
- `stale`（驗證過期、待重驗）→ **千萬不能排除**：它存在的意義就是提醒 agent 來重驗，藏掉＝把該看見的警訊一起遺忘、製造新洞。
- 節點內 `valid:false` 決策 → 歸 decisions 層（doctor 已審），不歸召回層。

降權桿當初把 `stale`+`superseded` 綁一起，是因為它只壓排序、無傷；升級成「排除」時**必須拆家**——**search 只濾 `superseded` 一種**。

## 借既有零件（citation 已按 r1 審計修正，零新機制）
1. doctor Check S 有 `superseded → skip` 機制——但它實際是 `status in ("stale","superseded")` **兩者一起 skip**（F1）。借它的「跳過」機制，**★絕不抄它把 stale 綁進去★**（那正是本案樞紐禁止的反例）。
2. `cmd_decisions --superseded` **不是** per-node 的「藏了再調出」開關（它預設就印全部決策、含 valid:false；`--superseded` 是另一個全庫掃描報表），所以「預設藏、明示調」的語意在它身上其實不存在（F5）。**只借「有個 opt-in 旗標」的命名先例**當 `--include-superseded` 的參照，不宣稱借到隱藏語意；`_is_forgotten` / `--include-superseded` 是**本計劃全新機制**。
3. `archive`（`ARCHIVABLE` 含 superseded）只覆蓋 `Verification/` 型節點（F6），範圍與本案不同；僅引用「遺忘要保留審計軌」的概念，不當實作佐證。
4. doctor **Check E1**（失效驗證 `dead_endorse`）＋ **Check E2**（建在被推翻決策 `build_on_dead`）是**兩個獨立 check**（r2 F7）——但 std Codex 證實它們**只管特定關係邊、並不覆蓋 superseded 節點的稽核**（見已知殘留 F2/F3）。∴ 不宣稱它們是本案的「審計半邊」，只借「有這類關係層 check」的存在事實，不當覆蓋佐證。

## ★明確不做（範圍刀）★
1. **impact 永不做預設藏**（F4 ＋使用者 2026-07-24 拍板）：impact 的 direct 命中＝「這節點直接引用你正在改的碼」＝**事故記憶**。一條作廢節點寫著「這樣改過、後來因 X 被打回」，正是 impact 最該遞給你的。預設藏＝把最有價值的警告消音。**impact 保持現狀。**
2. **context 推薦（`_reco_fused`）緩**（F3）：它與 impact 走 **BFS 多跳**。`A → 作廢 B → 活 C`（C 只能經 B 到達）時——只濾輸出：B 仍暗中撐 C 的圖分，遺忘不乾淨；連遍歷一起切：C 這個活節點被誤殺（比降權更硬的假陰性）。**先決定遍歷語意再做，本刀不碰。**
3. **B方案（重開降權桿 `_RANK_STATUS_MULT`）僅補充、非替代**：只降權不排除（強命中仍浮上），且重開需重跑 goldset。頂多對「明示召回出來」的 superseded 順手降權。（std F8：該常數凍結**預設** 1.0，但 `LUMOS_RANK_STATUS_MULT` 環境變數可覆寫——不是無條件零懲罰，評測時留意。）

## 已知殘留（std Codex 照出：「作廢」在全系統只實作一半，餵未來「要不要做全系統真遺忘」的決定）
本刀只補 search；下列是 Codex 跨家族審計照出的**其他半套處**，**本刀不做、但記下**免得被當「已解決」：
- **F1 context 基本鄰居也漏**：連 `context A`（不加 `--recommend`）都會印出鄰居的 status＋summary，作廢鄰居內容照樣遞出。context 的漏不只在 `_reco_fused` 推薦路——基本鄰居展開也算，一併留給 context 那刀。
- **F2 valid:false 決策 doctor 沒真接住**：search 只印命中行、不會連帶顯示該行所屬決策是 `valid:false`；doctor E2 只管有 `ended`＋連入邊＋`updated<ended` 的傳播邊，接不住「搜尋命中剛好落在死決策上」。∴「valid:false 歸 decisions 層已審」是**過度宣稱**，實為殘留。
- **F3 doctor 對作廢 Verification 不一致**：Check 3 只跳過 stale/fail、不跳 superseded（會要求把作廢驗證掛回）；E1 只把 stale/fail 當死背書；`sync-verified-by` 無 status 過濾（會主動補回作廢驗證）——與本刀「search 藏作廢」方向相反。doctor 側的作廢一致性是另一條線，本刀不碰。

## 測試策略（TDD，search-only）
1. **預設遺忘**：植 `status: superseded` 節點 → `lumos search <內文詞>` 預設**不含它**，且**印隱藏計數提示**。
2. **明示可召回**：同 query ＋ `--include-superseded` → **含它**。
3. **★樞紐回歸 stale 不被藏★**：植 `status: stale` 節點 → 預設 search **仍含它**。此案是 stale≠superseded 拆分的守門測試，**缺它坑就沒守住**。
4. **valid:false 決策不誤傷**：節點 status 正常、只內含一條 `valid:false` 決策 → 照常出現。
5. **★召回不退（前後對照，非 goldset gate）★（std F7）**：goldset 的 §6 gate 只比「ranked 對 legacy 的相對提升」，作廢節點若同時從兩邊被濾、相對提升不變、gate 照過——**擋不住「藏掉好答案」**。改用**改動前後對照**：造一個「作廢節點剛好是某 query 正解」的 fixture，量**改動前 vs 改動後**該 query 的召回——若掉了，就是把好答案藏掉了，測試必紅（逼你決定：該節點根本不該作廢，或該用 --include-superseded）。
6. **隱藏計數精確（std F6）**：藏 3 筆**命中結果**印 3、藏 0 印 0（不印）；庫裡有 100 篇 superseded 但 query 只命中 1 篇作廢 → 印「1」不是「100」（驗插點在命中確認後）。
7. **三路一致（r2 F2）**：`--legacy`／`--regex` 與預設 ranked 一樣排除 superseded、一樣有隱藏提示（濾網在命中確認後、分岔前）。
8. **輸出通道不汙染（r2 F3＋std F4/F5）**：`--json` 隱藏數走 `hidden_superseded` 欄位、stdout 仍合法 JSON；`--files-only` 檔名走 stdout、隱藏提示走 **stderr（照印，不取消）**；純 stdout 不混純文字提示。

## 風險面
- **blast radius ＝所有經 search 的召回**（agent 進場強制第一動作）。
- **goldset 交互**：排除會改召回指標 → 動 search **必重跑 goldset 對帳**（`retrieval-ranking` 評測合約）。
- **假陰性比降權硬**：「該看到卻沒看到」比「排序被壓」更傷 → 隱藏**必配** `--include-superseded` 逃生口 ＋ 印隱藏數（讓人知道有東西被藏）。

## 成本定位（過使用者 2026-07-24 立的尺）
正交（純機械過濾、非 LLM 判斷 loop）＋修正確性（非微小品質提升）＋便宜 → 屬「有人把關也該補」的機械正交項，與被緩置的 Mnemosyne 內容閘（跟人判斷重疊、貴）不同類。

## 實務隱患
- **stale 誤藏**：最大風險，靠測試案例 3（守門回歸）＋樞紐拆家紀律守。
- **靜默遺忘**：不印隱藏數＝使用者不知東西被藏 → 必印計數提示（案例 6）。
- **goldset 迴歸**：動排序/召回不重跑考卷＝盲改，違 `retrieval-ranking` 評測合約。
- **範圍蔓延**：context/impact 看似「順手一起做」，但各有 BFS／事故記憶的坑；範圍刀擋住，別偷偷擴。
