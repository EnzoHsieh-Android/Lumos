# 條款認領追溯_計劃

## 目標

解調研裁定②：計劃節點的規格條款只有升格成 ★INVARIANT★ 的被 Check T 管，其餘「要做 A/要處理 B」做沒做靠腦袋記。做輕量 RTM（需求追溯矩陣）：條款給 ID、驗證回指認領、指令掃無人認領。

PRIOR-ART: ① 最小解=一個唯讀掃描指令，無 schema 變更。② 世界解過：RTM/DO-178C requirements-based coverage、BDD 條款綁定；本 repo 已有 plan_refs 意圖鏈（Check 4）——本指令是它的條款級細化。③ 裁定=borrow-design。

## 變更規格

- **條款標記慣例**：計劃節點 body 的規格條款行內嵌 `[S1]`、`[S2]`…（`\[S\d+\]`；寫在「## 變更規格」等節的 bullet 上；不標=不追溯，opt-in）。
- **認領**：回指該計劃（plan_refs）的 Verification 節點 body 提及 `[SN]` 即認領（同一 ID 可多篇認領）。
- **指令** `lumos spec-trace <計劃節點> [--json]`（vault 內、唯讀）：列每條 ID 的認領狀態；輸出未認領清單。rc：全認領=0；有未認領=1（供 CI/人工判斷,不掛 pre-push）；節點無任何 `[SN]` 標記=0（opt-in 未啟用,印提示）。
- doctor 不加 check（v1）：opt-in 工具,避免對存量計劃掃出海量噪音。

## 測試計畫

- 合成 vault：計劃 3 條款、Verification 認領 2 → rc1 列 S3；補認領 → rc0；無標記計劃 → rc0+提示；plan_refs 沒回指的 Verification 提及 [SN] 不算認領（防偶然文字）。
