# 主動影響幅度偵測_計劃

> **狀態**:**已實作並通過 opus 終審**(branch feat/impact,634 passed)。設計經 `lumos-design-loop` 9 輪對抗審計(canary 9/9、抓 2 真算法 bug)→ writing-plans 12 task → subagent-driven TDD → opus whole-branch 終審(live 重現 C1/I1 2 blocker 真機修)。驗證見 [[2026-07-05_主動影響幅度偵測]]。

## 定位:自動修復的偵測地基

**本原語是 lumos 主打「關聯節點自動修復」的基礎建設。** `lumos impact --json` 的輸出不只餵給本計劃的 advisory 注入,更是**下游自動修復消費端的機器可讀輸入**——故 `--json` schema 必須**machine-actionable 且無歧義**:直接/間接分層、`hit`/`via` 邊型、`contract`/`combo` 標記、`node` rel 路徑,每一欄都要下游能不靠人判就消費。這抬高了 schema 精確度、邊集完整性(不漏 plan_refs/body-wikilink)、合約偵測正確性(IRREVERSIBLE 走對 RE)的標準——它們不是本計劃的錦上添花,是自動修復地基的承重牆。

## 背景:pull 非 push 的缺口

如今有 lumos 關聯節點輔助,維護某節點功能時**能不能主動偵測相關節點的影響幅度?** 查完圖譜結論:**影響幅度的資料完整建模了**(雙向 edges、合約→測試→Verification 鏈),**查詢原語也齊了**(`context`/`guard trace`/`backlinks`/`map`/`stale --candidate`)——**但「你正在改 X、這是波及面」不會自己跳出來**。

它靠兩件事驅動,都不是主動掃描:
1. **紀律**:CLAUDE.md「圖譜先行」*告訴* agent 動手前跑 `context`/`contracts`/`guard trace`——靠 agent 記得、誠實。
2. **機械閘**:pre-commit graph-sync gate *強迫*你碰圖譜——但它「改完了才擋」,且只判「有沒有碰圖譜」,**不算波及面**。

→ 它是 **pull(你去查)**,不是 **push(它來提醒)**。與記憶 `knowledge-sync-scatter-needs-mechanical-guard`(非圖譜節點)同病根:同步散落會漏、需機械守衛逼。本計劃補這個缺口。

## §1 架構與分工

沿用 lumos「Claude 編排、lumos 出原語」:

```
Claude 正要 Edit/Write/MultiEdit 一支 code
        ↓
[PreToolUse hook]  攔截 → 過濾(只 code 副檔名) → 查 session TTL 快取(這支檔近期注入過?)
        ↓ 未在冷卻窗內
  lumos impact --file <path> --json        ← lumos 出機械完整清單
        ↓
  hook 把清單 + 動手前分析指令 注入 session(additionalContext,不擋)
        ↓
Claude 動手前:判直接/間接節點會不會被這次改動波及、需不需同步 → 消噪 → 該改的改
```

- **lumos 只算、不判**:出「直接/間接」分層清單。
- **Claude 只判、不算**:拿清單做語意深度分析、消噪。
- **hook 只搬運 + 節流**:攔截、過濾、per-file TTL 快取、注入。

## §2 `lumos impact` 原語

**輸入**:`lumos impact --file <code路徑>`(**必填**)/ `--json` / `--depth N` / `--repo <root>`(r4-F5:hook 必傳,缺則退回 `find_vault`,見 §3)。
> r8-F3:**v1 只做 `--file <code路徑>` 一種入口**(hook 唯一用法);先前「也支援 `lumos impact <節點>` 給人手動查」**砍掉**(YAGNI)——節點→節點影響是不同算法(無 code→node 反查步驟),硬塞會留一整塊未定義分支。日後真要人手查節點鄰域,已有 `lumos context`/`map`/`backlinks` 覆蓋。

**演算法**(方案 A:即時、純記憶體、零落地索引):
1. **建 code→node 反查**:掃全節點 body inline-code 裡的 `code路徑`(複用 `_refcheck_scan` 的抽取)。
   - ⚠ **不掃 `core_refs`**(r7-F2,grep 實證):`core_refs` 是**跨 repo 節點指針**(doctor Check C `scripts/lumos:526-551` 讀作 `repo_root.parent/<seg>` 的 core-knowledge `.md` 節點路徑),**不是 code 引用路徑**——拿它比對「被改動的 code 檔」永遠不命中。code→node 反查**只靠 body inline-code**。(前 6 輪一直把 core_refs 誤當 code 反查來源,r7 修正。)
   - ⚠ **`_refcheck_scan` 的 `top_dirs` 過濾**(r7-F6):`_refcheck_scan`(`scripts/lumos:3944-3948`)會**靜默跳過**首段不在 repo 頂層目錄集的 token(`token.split("/")[0] not in top_dirs`)。故 body 裡寫非頂層開頭的 code 路徑不會被反查到——這是機械限制,列入 §4 天花板。
   - ⚠ **不能直接複用現有 `load_vault` 記憶體**(r1-F1):`Note.__slots__`(`scripts/lumos:160`)**不存 body**;且 `load_vault:184` 在算 `targets` *之前* 就用 `INLINE_CODE_RE.sub("","")` **把 inline-code 整段刪掉**——記憶體裡連反查原料都沒有,`_refcheck_scan(text,...)` 吃的又是原始 text 字串。→ **實作二選一(spec 選前者)**:(a) impact 呼叫時**重新讀盤每個 .md** 把 text 餵 `_refcheck_scan`(即時、零落地索引成立,符合方案 A);(b) 擴充 `Note` 存 body inline-code spans。選 (a):200 篇讀盤成本無感、不動 Note class。
   - ⚠ **路徑規範化**(r3-F5):hook 傳來的是**絕對路徑**(`/Users/.../scripts/lumos`),但節點 body/`core_refs` 記的是 **repo-relative token**(`scripts/lumos`)。反查前須把絕對路徑轉 repo-relative(`os.path.relpath(abs, repo_root)`)再比對,否則永遠 miss。`_refcheck_scan` 需 `repo_root` 參數(見 §3 hook 傳 `--repo`)。
2. **直接關聯(1-hop)** = 反查命中的節點。
3. **間接關聯** = 從直接節點集當種子沿邊 BFS 展開。**hop 語意**(r8-F5):hop = 距最近直接節點的 BFS 距離,`hop ∈ 1..depth`(hop 1 = 直接節點的鄰居);`depth`(預設 2,見 §3 config)= 最大 hop。(廢除舊「2..depth」記號——那與 schema 的 hop=1 矛盾。)
   - ⚠ **visited/cycle guard**(r8-F4):BFS 必須維護 `seen` set(**種入全部直接節點**),已在 direct 或已訪的節點**不再進 indirect**(lumos 圖譜 related 常互指、A↔B 環正常;複用 `map` 的 `seen` 模式 `scripts/lumos:908-923`)。否則直接節點會沿環重入 indirect,下游自動修復讀到「同節點既 direct 又 indirect」矛盾資料。
   - ⚠ **雙向 direction 衝突**(r8-F7):若 dest 同時經 `out_e`(outlink)與 `in_e`(backlink)可達 → **標 `backlink` 優先**(反向依賴=改 source 波及 dest,自動修復最該看)。
   - ⚠ **邊集分兩路**(r1-F2 + r2-F1):`env.edges`(`scripts/lumos:251-263`)由 `n.targets` 建,而 `n.targets` 只收 **wikilink `[[..]]` 格式**且**不含 summary block scalar 內的**(`load_vault:186-197` 依鐵則 2 整段排除 block、`fullmatch [[..]]` 濾非 wikilink)。故:
     - **`related` / `verified_by` / `plan_refs`(YAML list wikilink)+ body 內的 `[[..]]`**(r5-F5)→ 全都真在 `env.edges`(`n.targets = extract_targets(body_clean) + fm_targets`,body wikilink 與 plan_refs 都入),**雙向**含 backlinks,直接走 BFS。BFS 別只過濾 related/verified_by、漏掉 plan_refs 與 body wikilink 邊。
     - **`core_refs`**(r2-F1:純文字路徑非 wikilink,`scripts/lumos:526-551` 讀作跨 repo 節點路徑)→ **不在 `env.edges`**,且目標節點**住在 core-knowledge repo、不在本 `env.notes`**(r7-F2)。故:直接節點若有 `core_refs`,**把該 core 節點當「不可展開的跨 repo 葉」輸出**(標 `via:core_refs`、`cross_repo:true`、`no_expand:true`)——**不試圖 BFS 進它**(`env.notes[dest]` 會 miss)。讀法 `as_list(n.fields.get("core_refs"))`(r3-F8:值可能 str/list,直接 iterate 字串會拆單字元)。
     - **summary 的 `DEP:[[..]]`/`VERIFY:[[..]]`** 語意邊 → **不在 `env.edges`**,列為獨立後續工作項(需另建 summary-line 解析器繞過鐵則 2),v1 明列不涵蓋。
   - ⚠ **到達路徑標記需新 infra**(r1-F5):`env.edges` 只存 `(source,dest)` pair、**無 edge type**。要標 `via`,**選 BFS-time 二次反查**(r4-F4),**含 wikilink 所在端**(r9-F1,關鍵——`env.edges` 是 `out_e[a].append(b)`/`in_e[b].append(a)`,wikilink 永遠寫在「連出去」那一端):
     - **outlink**(dest ∈ `out_e[frontier]`):wikilink 在 **frontier** → 讀 `frontier.fields` 找指向 dest 的 wikilink → 該欄位即 `via`。
     - **backlink**(dest ∈ `in_e[frontier]`):wikilink 在 **dest**(dest 連向 frontier)→ 讀 **`dest.fields`** 找指向 frontier 的 wikilink → 該欄位即 `via`。(**別讀 frontier.fields**——它不含指向 dest 的連結,會全 miss。)
     - 比對走既有 `link_target()`+`env.resolve()` 把 `"[[stem]]"` 轉 rel 再比(r6-F6,同全 lumos pattern)。
     - **body wikilink fallback**(r5-F3):`n.fields` 只有 frontmatter,若在「wikilink 所在端」的任何 frontmatter 欄位都找不到 → 該邊來自 body(`extract_targets(body_clean)`)→ 判 `via:body-wikilink`。
     - **不改 `env.edges` 建構**(全 lumos 共用核心 infra);二次反查只在 impact 自己 BFS 內做、隔離。
4. **排序規則**:合約節點(★INVARIANT★/★IRREVERSIBLE★)優先 → 直接關聯優先於間接 → 同層依 hop 距離升序。每筆標:**層級**(直接/間接第幾 hop)+ **到達路徑**(直接=命中特徵 `hit`;間接=邊型 `via`,見 §2 schema)+ **是否帶合約**。
   - ⚠ **合約偵測分兩軸**(r6-F3):`★INVARIANT★`/`★DEBT★` 走 `extract_contracts`(`scripts/lumos:934`,只掃這兩者);`★IRREVERSIBLE★` **不在** `extract_contracts`、走獨立 `IRREVERSIBLE_RE`(`:1184`,可逆性軸)。impact 要拿到 `IRREVERSIBLE` 合約標記須**另外用 `IRREVERSIBLE_RE` 掃 summary**,只調 `extract_contracts` 永遠拿不到。

**輸出範例**(人讀,與下方 `--json` 同一組資料;depth=2):
```
改動 scripts/lumos (由讀盤 refcheck 抽取)
── 直接關聯 (2) ──
  ⚠合約 Systems/lumos-refcheck   ★INVARIANT★★COMBO★ (body inline-code 命中)
        Systems/lumos-cli-read              (body inline-code 命中)
── 間接關聯 (2) ──
  hop1  Systems/pitfalls-lint-adapter   backlink ← via related ← lumos-cli-read
  hop1  core-knowledge/systems/rule     outlink ← via core_refs ← lumos-cli-read [跨repo葉,不展開]
```
> 註(r1-F4):本圖譜的 `lumos-refcheck.md` **無 `core_refs` 欄位**;它被反查命中是靠 body inline-code 的 `` `scripts/lumos` ``——這也印證 code→node 反查只認 body inline-code(r7-F2)。

**`--json` schema**(r2-F7;下游自動修復消費端的機器可讀契約——每欄型別/可空/枚舉/presence 皆定死,不靠人判):
```json
{
  "file": "scripts/lumos",
  "direct": [
    {"node": "Systems/lumos-refcheck", "hit": "body-inline-code", "contract": "INVARIANT", "combo": true},
    {"node": "Systems/lumos-cli-read", "hit": "body-inline-code", "contract": null, "combo": false}
  ],
  "indirect": [
    {"node": "Systems/pitfalls-lint-adapter", "hop": 1, "via": "related",
     "direction": "backlink", "from": "Systems/lumos-cli-read", "contract": null, "combo": false},
    {"node": "core-knowledge/systems/rule", "hop": 1, "via": "core_refs",
     "direction": "outlink", "from": "Systems/lumos-cli-read",
     "cross_repo": true, "no_expand": true, "contract": null, "combo": false}
  ]
}
```
（範例:兩筆 indirect 皆 `hop:1`=直接節點 `lumos-cli-read` 的鄰居,`from` 指該直接節點,hop 語意一致。）
- **欄位定義**(每筆物件):
  - `node`(str,必有):vault rel 路徑;**跨 repo 葉例外** = core-knowledge 節點路徑(`cross_repo:true` 時)。
  - `direct.hit`(str,必有,r3-F3)= 反查命中特徵,**枚舉只 `body-inline-code`**(r7-F2:core_refs 已不做 code 反查,故 hit 不再有 `core_refs`)。
  - `indirect.via`(str,必有)= BFS 邊型,**完整枚舉 5 種**(r5-F5/r7-F5,一處列全):`related`/`verified_by`/`plan_refs`/`body-wikilink`/`core_refs`。
  - `indirect.direction`(str,必有,**r7-F1**)= `backlink`(dest 連向 source,即改 source 波及 dest 的反向依賴,自動修復最該看)或 `outlink`(source 連向 dest)。從 `env.edges` 的 `in_e`/`out_e` 哪側取得決定。`direct` 無此欄。
  - `indirect.hop`(int,必有)= 距最近直接節點 BFS 距離,`1..depth`;`indirect.from`(str,必有,r8-F6)= **BFS 路徑上前一跳(近一層)節點的 rel 路徑**——hop=1 時 `from` 是某直接節點,hop=k 時是 hop=k-1 的節點(非固定 direct)。`direct` 無 `hop`/`from`。
  - `cross_repo`/`no_expand`(bool,r7-F2)= **僅 `via:core_refs` 的跨 repo 葉有**(true);其餘筆**不含這兩 key**(下游用 `.get(k, false)`)。
  - `contract`(str|null,必有)∈ `INVARIANT`/`IRREVERSIBLE`/`null`(取最高:`IRREVERSIBLE`>`INVARIANT`)。
  - `combo`(bool,**必有、每筆都出、無 ★COMBO★ 則 `false`**,r3-F6/r4-F6/r7-F3):`★COMBO★` 是 `★INVARIANT★` 同行後綴修飾非獨立 marker(偵測邏輯 `scripts/lumos:725` 的 `"★COMBO★" not in inv`,r9-F6:`:718` 只是 Check K 段首註解非偵測碼),故不入 `contract` enum、走獨立布林。

  - 再次編輯同檔:距上次注入 **< TTL → 壓掉**(壓「連續 iterate 存檔洗版」);**≥ TTL → 重新注入**(長 session 隔久回頭、脈絡已遠,復活)。預設 TTL **20 分**(見下方 config)。

  - (可選加碼:影響集內容變了無視 TTL 立即重注;主要修正仍是 TTL。)
- **注入方式**:`{"hookSpecificOutput": {"hookEventName": "PreToolUse", "additionalContext": "<清單+指令>"}}`,**永不 block、永不改 permissionDecision**。

- **空集合**:`lumos impact` 回空 → 不注入(不洗版)。
- **注入的指令文字**(附清單後):「動手前先判上列直接/間接節點會不會被你這次改動影響、需不需要同步更新圖譜。消掉不相關的,對真正受影響的:該同步就同步,不確定就記一句。」
- **fail-open**(依上方 rc 協定):`lumos` 不在 PATH / `lumos impact` 內部出錯(rc 非 0 且非 3)→ **純靜默放行**;非圖譜專案 / vault 找不到(rc 3)→ **印一行 debug 後放行**(可察覺,不混入純靜默)。任何情形**絕不擋 Claude 幹活**。

## §4 誠實天花板(收斂後向人講明)

1. **只抓「圖譜 body 寫下 code 路徑」的節點**(r8-F1 同步 r7-F2):某節點跟這支 code 有關但從沒在 **body inline-code** 寫下該路徑 → 反查看不到(GIGO,同 refcheck)。(`core_refs` 不算——它是跨 repo 節點指針非 code 路徑。)影響偵測完整度 = 圖譜 body code 引用的完整度。
2. **只在 Claude session 內生效**:人手動改 code 不觸發(PreToolUse 限定)。選 trigger 時明確接受的範圍。
3. **「Claude 分析過」≠「影響真的處理了」**:注入只保證 Claude *看到*、被要求判;判得對不對、有沒有真同步是沒閉合的迴歸(同 design-loop 整合性天花板)。它是**可觀測 + 摩擦 + push**,不是 oracle。
4. **新建檔(Write 新路徑)幾乎必回空集合**(r4-F9):圖譜還沒有節點引用尚不存在的檔 → 反查回空、不注入。這是**預期行為非缺陷**(新檔的影響要等它被寫進圖譜/被引用後才算得出)。
5. **body inline-code 的 `top_dirs` 靜默過濾**(r7-F6):`_refcheck_scan` 只認首段落在 repo 頂層目錄集內的路徑 token;body 若把 code 路徑寫成非頂層開頭(或相對怪路徑),不會被反查到、也不報錯。屬「只抓圖譜寫對路徑的」的細機械限制。
