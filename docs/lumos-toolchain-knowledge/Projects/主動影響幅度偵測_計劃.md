# 主動影響幅度偵測_計劃

> **狀態**:設計定稿(2026-07-05 brainstorming),待 `lumos-design-loop` 過審 → writing-plans → 實作。

## 背景:pull 非 push 的缺口

如今有 lumos 關聯節點輔助,維護某節點功能時**能不能主動偵測相關節點的影響幅度?** 查完圖譜結論:**影響幅度的資料完整建模了**(雙向 edges、合約→測試→Verification 鏈),**查詢原語也齊了**(`context`/`guard trace`/`backlinks`/`map`/`stale --candidate`)——**但「你正在改 X、這是波及面」不會自己跳出來**。

它靠兩件事驅動,都不是主動掃描:
1. **紀律**:CLAUDE.md「圖譜先行」*告訴* agent 動手前跑 `context`/`contracts`/`guard trace`——靠 agent 記得、誠實。
2. **機械閘**:pre-commit graph-sync gate *強迫*你碰圖譜——但它「改完了才擋」,且只判「有沒有碰圖譜」,**不算波及面**。

→ 它是 **pull(你去查)**,不是 **push(它來提醒)**。與記憶 `knowledge-sync-scatter-needs-mechanical-guard`(非圖譜節點)同病根:同步散落會漏、需機械守衛逼。本計劃補這個缺口。

## §1 架構與分工

沿用 lumos「Claude 編排、lumos 出原語」:

```
Claude 正要 Edit/Write 一支 code
        ↓
[PreToolUse hook]  攔截 → 過濾(只 code 副檔名) → 查 session TTL 快取(這支檔近期注入過?)
        ↓ 未在冷卻窗內
  lumos impact --file <path> --json        ← lumos 出機械完整清單
        ↓
  hook 把清單 + 動手前分析指令 注入 session(additionalContext,不擋)
        ↓
Claude 動手前:判直接/間接節點會不會被這次改動波及、需不需同步 → 消噪 → 該改的改
```

- **lumos 只算、不判**:出「直接/間接」分層清單。
- **Claude 只判、不算**:拿清單做語意深度分析、消噪。
- **hook 只搬運 + 節流**:攔截、過濾、per-file TTL 快取、注入。

## §2 `lumos impact` 原語

**輸入**:`lumos impact --file <code路徑>`(也支援 `lumos impact <節點>` 給人手動查);`--json` / `--depth N`。

**演算法**(方案 A:即時、純記憶體、零落地索引):
1. **建 code→node 反查**:掃全節點的 ① frontmatter `core_refs`(純文字路徑,見 `n.fields`)② body inline-code 裡的 `code路徑`(複用 `_refcheck_scan` 的抽取)。
   - ⚠ **不能直接複用現有 `load_vault` 記憶體**(r1-F1):`Note.__slots__`(`scripts/lumos:160`)**不存 body**;且 `load_vault:184` 在算 `targets` *之前* 就用 `INLINE_CODE_RE.sub("","")` **把 inline-code 整段刪掉**——記憶體裡連反查原料都沒有,`_refcheck_scan(text,...)` 吃的又是原始 text 字串。→ **實作二選一(spec 選前者)**:(a) impact 呼叫時**重新讀盤每個 .md** 把 text 餵 `_refcheck_scan`(即時、零落地索引成立,符合方案 A);(b) 擴充 `Note` 存 body inline-code spans。選 (a):200 篇讀盤成本無感、不動 Note class。
2. **直接關聯(1-hop)** = 反查命中的節點。
3. **間接關聯(2..depth)** = 從直接節點沿邊 BFS 展開到 `depth`(預設 2,見 §3 config)。
   - ⚠ **邊集分兩路**(r1-F2 + r2-F1):`env.edges`(`scripts/lumos:251-263`)由 `n.targets` 建,而 `n.targets` 只收 **wikilink `[[..]]` 格式**且**不含 summary block scalar 內的**(`load_vault:186-197` 依鐵則 2 整段排除 block、`fullmatch [[..]]` 濾非 wikilink)。故:
     - **`related` / `verified_by`**(YAML list wikilink)→ 真在 `env.edges`,**雙向**含 backlinks,直接走 BFS。
     - **`core_refs`**(r2-F1:是**純文字路徑**非 wikilink,見 §1、doctor Check C `scripts/lumos:526-551` 直接當檔案路徑讀)→ **不在 `env.edges`**;要沿它展開須**另讀 `n.fields["core_refs"]` 手動補邊**(單向,指向 core-knowledge 跨 repo 節點)。
     - **summary 的 `DEP:[[..]]`/`VERIFY:[[..]]`** 語意邊 → **不在 `env.edges`**,列為獨立後續工作項(需另建 summary-line 解析器繞過鐵則 2),v1 明列不涵蓋。
   - ⚠ **到達路徑標記需新 infra**(r1-F5):`env.edges` 只存 `(source,dest)` pair、**無 edge type**。要標「← via related ← X」需在建邊時同記來源欄位,或 BFS 時對命中節點做第二次欄位反查。列為本原語要新建的部分,非「複用」。
4. **排序規則**(r1-F3):合約節點(★INVARIANT★/★IRREVERSIBLE★)優先 → 直接關聯優先於間接 → 同層依 hop 距離升序。每筆標:**層級**(直接/間接第幾 hop)+ **到達路徑**(沿哪條 frontmatter 欄位/哪個方向)+ **是否帶合約**。

**輸出範例**(`--json` 給 hook;預設人讀):
```
改動 scripts/lumos (由讀盤 refcheck 抽取)
── 直接關聯 (2) ──
  ⚠合約 Systems/lumos-refcheck   ★INVARIANT★ ...(body inline-code 路徑命中)
        Systems/lumos-cli-read   (body inline-code 路徑命中)
── 間接關聯 (1, ≤2-hop) ──
        Systems/pitfalls-lint-adapter   ← via related ← lumos-cli-read
```
> 註(r1-F4):本圖譜的 `lumos-refcheck.md` **無 `core_refs` 欄位**;它被反查命中是靠 body inline-code 的 `` `scripts/lumos` ``,不是 core_refs。`core_refs` 是**跨 repo 指針**(指向 core-knowledge),命中屬少數情形。

**`--json` schema**(r2-F7,hook 消費此格式):
```json
{
  "file": "scripts/lumos",
  "direct":   [{"node": "Systems/lumos-refcheck", "via": "body-inline-code", "contract": "INVARIANT"}],
  "indirect": [{"node": "Systems/pitfalls-lint-adapter", "hop": 2, "via": "related", "from": "Systems/lumos-cli-read", "contract": null}]
}
```
`via` ∈ `body-inline-code`/`core_refs`/`related`/`verified_by`;`contract` ∈ `INVARIANT`/`IRREVERSIBLE`/`null`;`direct` 無 `hop`/`from`。

**方向語意**:改 X 時「誰受影響」= 主要是**反向依賴**(誰連到我 / backlinks),但 lumos 不替你判方向——**全撈、標方向**,交 Claude 判(符合「全撈、AI 消噪」原則)。

**方案取捨**(為何 A):落地索引(B)引入 staleness + 多一個會漂移的 artifact;只複用 guard trace(C)漏掉沒綁合約的節點,與「不能只看直接關聯」的把關意圖矛盾。lumos 本來就全圖載入記憶體(200 篇無感),A 永遠新鮮、零維護。

## §3 PreToolUse hook

- **matcher**:`Edit|Write|MultiEdit`(r1-F7:**不含 NotebookEdit**——lumos-toolchain 無 notebook;若未來要涵蓋須顯式加進 matcher,不靠括號註解)。
- **過濾**:讀 hook input 的 `file_path` → 只對 code 副檔名跑(跳過 `docs/*-knowledge/`、`node_modules`、`bin`、`obj`、`__pycache__` 等)。非 code → 直接放行、零動作。
  - ⚠ **CODE_EXTS 來源**(r1-F6):現有三份 CODE_EXTS **不一致**——`scripts/hooks/claude/check-graph-sync.py:24`(Python set,20 副檔名)、`scripts/hooks/pre-commit:67`(bash 正則,同 20)、`scripts/lumos` `CODE_EXTS_T`(Python set,**僅 12**,缺 `.kts/.mjs/.cc/.cpp/.h/.swift/.c/.hpp`)。新 hook 是 Python,**單一源取 `check-graph-sync.py` 的 CODE_EXTS(20 版)**——但 `check-graph-sync.py` 是獨立腳本非 package(無 `__init__.py`)、**無法 import**(r2-F8),故**抄一份到 hook 並在兩處留「同源 check-graph-sync.py」註解**;順帶記一筆:`lumos CODE_EXTS_T` 的 12-vs-20 分歧屬既有債,本計劃不順手改(避免擴大範圍)、只在此標記。
- **per-file 節流(TTL 冷卻窗)**:標記檔 `<tmpdir>/lumos-impact-<session_id>/<filepath hash>` 記時間戳。`session_id` 取 hook payload 的 `session_id` 欄位(UUID,天然路徑安全,無需截斷/hash)(r2-F6)。
  - **跨平台 tmpdir**(r1-F10):用 `tempfile.gettempdir()`(Windows→`%TEMP%`、Linux→`/tmp`、macOS→`$TMPDIR`),**不寫死 `$TMPDIR`**(本專案有 Windows 支援)。
  - **hash 演算法**(r1-F11):`hashlib.sha1(filepath_abs.encode()).hexdigest()[:16]`(定死,不留實作者自選)。
  - 再次編輯同檔:距上次注入 **< TTL → 壓掉**(壓「連續 iterate 存檔洗版」);**≥ TTL → 重新注入**(長 session 隔久回頭、脈絡已遠,復活)。預設 TTL **20 分**(見下方 config)。
  - **為何 TTL 不是 per-session 一次**:使用者習慣單一長 session 保脈絡。「session 終身一次」會讓 turn 90 回頭改同檔時、turn 5 的注入早被捲走/壓縮 → 第二次修改變**啞巴**,偏偏那是提醒最該出現時。TTL 把「連續 iterate(靜音)」與「隔久回頭(復活)」乾淨分開。
  - (可選加碼:影響集內容變了無視 TTL 立即重注;主要修正仍是 TTL。)
- **注入方式**:`{"hookSpecificOutput": {"hookEventName": "PreToolUse", "additionalContext": "<清單+指令>"}}`,**永不 block、永不改 permissionDecision**。
- **空集合**:`lumos impact` 回空 → 不注入(不洗版)。
- **注入的指令文字**(附清單後):「動手前先判上列直接/間接節點會不會被你這次改動影響、需不需要同步更新圖譜。消掉不相關的,對真正受影響的:該同步就同步,不確定就記一句。」
- **fail-open**:`lumos impact` 出錯 / lumos 不在 PATH / 非圖譜專案 → 靜默放行(絕不擋 Claude 幹活)。
- **config**(r1-F8):設定合入既有 `.lumos/config.json`(`scripts/lumos:1037` 的專案配置入口,已有 `test_profile`/`lint`/`compose-metrics` 等)新增 `impact` key(`{"impact": {"depth": 2, "ttl_min": 20}}`),**不新增 `.lumos/impact.json`**(避免 `.lumos/` config 分裂)。缺 key → 用預設 depth=2/ttl=20min。

## §4 誠實天花板(收斂後向人講明)

1. **只抓「圖譜寫下路徑」的節點**:某節點跟這支 code 有關但從沒把路徑寫進 `core_refs`/body → 反查看不到(GIGO,同 refcheck)。影響偵測完整度 = 圖譜 code 引用的完整度。
2. **只在 Claude session 內生效**:人手動改 code 不觸發(PreToolUse 限定)。選 trigger 時明確接受的範圍。
3. **「Claude 分析過」≠「影響真的處理了」**:注入只保證 Claude *看到*、被要求判;判得對不對、有沒有真同步是沒閉合的迴歸(同 design-loop 整合性天花板)。它是**可觀測 + 摩擦 + push**,不是 oracle。

## §5 測試

- **原語單元**(`scripts/test_lumos.py`):反查索引從 fixture 正確建 code→node;`--depth` 邊界(1 只出直接、2 出間接);直接/間接分層 + 到達路徑 + 方向標記正確;未知檔回空;合約節點被標並排前;`--json` schema。
- **hook**:code 副檔名才跑、非 code 放行、TTL 冷卻窗(窗內壓/窗外復活)、空集合不注入、lumos 缺席 fail-open 不擋。
- **anchor**:hook 檔進 anchor 完整性機制。⚠(r1-F9)`ANCHOR_FILES`(`scripts/lumos:3818-3824`)是**硬碼 list**——實作時須手動把新 hook 檔加進該 list 並跑 `lumos anchor approve`(非自動)。
