# 狀態標籤同步守衛_計劃

## 背景

2026-07-20 盤點時發現「進行中」搜尋混入大量已完成計劃：16 個節點的 `status` 欄位已更新、`tags` 裡的 `status/*` 標籤停在舊值。根因是資料重複存兩處（欄位＋標籤）而寫入工具只改一處——`lumos new` 模板生出兩處，`lumos set status` 只動欄位；tags 是 list，lumos 又只有 `append` 沒有移除，正常流程下標籤永遠不會被更新。本次 16 處已手動清零（16 files, ±1 行），但不堵路下次還會長回來。

PRIOR-ART: 最小解在既有機制層（`cmd_set` 單函式小修＋lint 加一條 check，無新機制）；世界解過——Obsidian 社群共識是 tag/property 擇一別重複存（見下方來源），DB 界對必要的反正規化用「觸發器同步＋完整性檢查」；裁定＝borrow-design（保留既定並存慣例，借同步＋守衛形狀原生實作）。

## 規格

- **[S1] 同步觸發器**：`cmd_set`（`scripts/lumos:4677`）當 `key == "status"` 時，同步把 tags list 中既有的 `status/*` 項改寫為 `status/<新值>`。無 `status/*` 標籤的節點不添加（純同步、不發明）；寫後自驗（`atomic_write_verify`）同時驗欄位與標籤。
- **[S2] 漂移守衛**：`lumos lint`（doctor 併掃）新增檢查——`status` 欄位與 tags 內 `status/*` 標籤不一致即報錯（硬擋，同鐵則 lint 等級）。守衛兜住所有繞過 S1 的寫入路徑（手改、外部工具）。
- **非目標**：通用 list 項移除指令（`untag`/`set --replace`）緩議——本輪只堵實際漏過的 status 路；「砍標籤徹底去重」的遷移不做（檢索對 fm 標籤的依賴未驗證、動 140 篇成本不成比例）。

## 替代方案（已考慮）

1. **徹底去重（世界標準解）**：刪全 vault `status/*` 標籤、模板不再生。否決：需先驗證檢索/既有查詢對標籤的依賴，遷移面大，收益只是省一條同步邏輯。
2. **只加守衛不加同步**：lint 報錯後叫人手修。否決：工具自己寫入造成的漂移叫人擦屁股，違反「機械守衛」哲學。

## 驗收

1. 單測：`set <node> status done` 後，該節點 tags 內 `status/*` 同步為 `status/done`；無 status 標籤的節點不受影響。
2. 單測：lint 對人造 mismatch 節點報錯、一致節點通過。
3. 全 vault 掃描 0 漂移（現況已清零，守衛上線後維持）。
