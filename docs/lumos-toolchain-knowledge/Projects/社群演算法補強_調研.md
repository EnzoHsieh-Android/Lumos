# 社群演算法補強_調研

2026-07-10 deep-research 調研：對圖譜已知四缺口（a. canary 對技術密集 spec 易漏、b. 自主 loop 只長不剪、c. 知識同步散落、d. 散文段審計空轉）搜學界/GitHub 可借鑑演算法。方法：5 路搜索角度 → 24 來源 → 118 claims → 每條 3 票對抗查證 → 19 存活 / 6 被殺。**全部裁定為 borrow-design**（理論/設計借用、stdlib 原生實作，符合零依賴家規）。

PRIOR-ART: 本節點即「先問世界」的系統性存查——mutation testing/IBIR/FLAWS/LLM-judge 可靠度/co-change mining 五支文獻線已真搜真驗，非憑印象。

## 存活結論（機制 × 可借鑑理論 × 借法）

### 1. canary loop × mutation testing 理論（高信度，3-0）
- 來源：Papadakis et al. survey（Advances in Computers 2019）、Just et al. FSE 2014。
- 植缺陷驗審計員的做法有理論根基（讓「審計有效性」可證偽）；73% 真實缺陷與常用 mutant 耦合。
- **caught-rate 三修正**：① 只有近滿分的 caught-rate 才有強訊號（中段分數相關弱）→ 收斂閘要 near-perfect 不是門檻分；② 冗餘/過易 canary 會灌水指標（冗餘識別不可判定）→ 需篩掉「殺 A 必殺 B」的重複難度 canary；③ ~17-27% 真實缺陷類型（演算法級改動、刪除類）天生植不出來 = canary 理論天花板 → missed 類別要分類追蹤、用實作真測補（與圖譜既有結論 [[Systems/canary-audit]] 天花板一致，現在有文獻背書）。

### 2. canary 生成硬化：FLAWS pipeline（中信度，preprint）
- 來源：FLAWS benchmark（arXiv 2511.21843，2025-11）。
- 三步生成：抽 spec 核心可證偽 claim → 生成使其失效的概念級錯誤（局部邏輯自洽）→ **difficulty filter：丟棄生成模型自己就能抓到的**（存活率 7-39%）。
- 同 benchmark 實證 frontier LLM 對技術密集文件植錯的漏檢率極高（top-10 猜測都 <50% 準確）→ 支持 missed-rate 當一級指標（呼應 memory「canary 可信度因 spec 而異」）。

### 3. canary 寫實化：IBIR 模式接 pitfall_when 語料（中信度）
- 來源：IBIR（ACM TOSEM 2022，github.com/serval-uni-lu/IBIR）。
- 「bug report → 定位 → 修復 pattern 反轉植錯」比盲目 mutation 寫實；其偵測率能區分有效/無效 test suite。
- 借法：把 Issues 事故語料的「修法」反轉成 canary 模板、植在 pitfall_when 命中的 spec 段落——直接接既有 [[Systems/pitfalls-code-loop]] 事故機制（④事故語料擱置線的新出口）。
- ⚠ arXiv 版「50x 耦合率」被查證否決（TOSEM 版下修 ~9x），引用時勿用該數字。

### 4. 審計 loop 架構裁定：panel+meta-judge > 多輪互辯（中信度）
- 來源：Judging with Many Minds（EMNLP 2025 Findings）、PoLL（Cohere 2024，preprint 有利益衝突）。
- multi-agent debate 第一輪就劇烈放大 position/verbosity/CoT/bandwagon 偏誤且後續不自癒；meta-judge 聚合多個一級判決明顯更抗偏誤；panel 異家族（disjoint model families）intra-model bias 更低。
- 借法：code-loop/design-loop reviewer 結構 =「N 個獨立、異家族、互不通訊 reviewer + meta-judge 收斂裁定」，**不要讓 reviewer 看彼此輸出迭代辯論**。與既有 [[Systems/heterogeneous-finder-ensemble]]、[[Systems/cross-family-audit]] 方向一致，文獻加碼「禁互辯」這條。
- ⚠ 範圍限定：debate 放大的是「偏誤」；另有 NeurIPS 2025 研究稱 debate 提升「準確率」——兩軸分開看。

### 5. 單輪審計不可當定論：多 run 多數決 + conformal 區間（中信度）
- 來源：Rating Roulette（EMNLP 2025 Findings）、conformal LLM-judge（EMNLP 2025 main）。
- 同 judge 同輸入跨 run 的 Krippendorff's α 最好僅 0.563（<0.8 可靠門檻）；多 run 多數決改善與人類一致（但只壓 stochastic 變異、不壓 correlated 系統性盲點——後者要靠異家族 panel）。
- split conformal prediction：held-out 校準集 + 分位數即可給判決不確定性區間，純 stdlib 可實作。
- 借法：canary 審計判決 ≥3 獨立 run 取多數決；累積人工查證過的審計結果當校準集。
- ⚠「conformal 無條件 coverage 保證」被否決（需 exchangeability、原論文靠 logprobs）。

### 6. 知識同步散落守衛：co-change mining（高信度，3-0；直答缺口 c）
- 來源：Zimmermann et al. ROSE（IEEE TSE 2005，foundational MSR）。
- git 歷史挖關聯規則能揭露靜態分析看不到的 code↔非code 耦合（論文例：fKeys[] 11 改 10 次與 plugin.properties 同改）；commit 前警告不完整改動。
- 借法：stdlib 解析 git log → 挖「A 改 ⇒ B 歷史 X% 同改」規則 → pre-commit/doctor check：改了機制文件沒碰歷史上總同改的列舉表就警告。直答 memory「知識同步散落會漏、需機械守衛」。
- ⚠ 量級限定：粗粒度 top-3 >70% 正確、細粒度僅 15-26%；小 repo 共改歷史夠不夠出有意義 confidence 未驗證（開放題）。

## 無存活結論的子題（不代表文獻不存在）
- 缺口 b 機制修剪/deletion 治理（dead code reaping 類比）、KG 健康度指標（centrality/link prediction）、metamorphic testing、輕量增量影響分析——本輪查證無存活 claim，需另做針對性搜尋。

## 候選落地優先序（未裁定，待討論）
