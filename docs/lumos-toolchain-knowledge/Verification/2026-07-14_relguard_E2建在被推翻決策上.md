# 驗證：doctor Check E2 建在被推翻決策上（+ P1）

關係層傳播守衛 phase-2（計劃 [S7] + 前置 [P1]）落地：補網再添一道**時序關係檢查**，把 Check 4 的「結案後有較新驗證」時序 proxy 推進到「**決策翻案**」層——某節點決策被推翻後，指向它的 `verified_by`/`plan_refs` 來源若沒跟上（`updated` 早於決策 `ended`）→ 標「可能建在被推翻決策上」。

## 變更範圍
- `scripts/lumos` cmd_doctor：新增 `Check E2`（放 Check E1 之後、Check H 之前）。**內聯 typed 反查**（掃全圖節點的 `verified_by`/`plan_refs` 具名欄位建 `target→[(source,edge_type)]`，不建 [P0] 可重用 index——doctor 本就掃全圖，補網不需；主網 proactive 連鎖才需 P0）。每個 `valid:false` 且有 `ended` 的決策 → 其 typed 連入來源 `updated < ended` → warn_soft。無 ended 的被推翻決策跳過。gov 記 `check-e2`。
- `scripts/lumos` cmd_decision_supersede（**[P1]**）：`ended` 缺則自動補今日 + 同步 bump 頂層 `updated`，**同一份 fm、單次 `atomic_write_verify`**（不串第二個寫命令、避免半完成）。
- `scripts/test_lumos.py`：`t_check_e2_build_on_superseded`、`t_decision_supersede_p1_autofill`。

## 天花板
- **時序法是啟發**：`updated < ended` 只是「可能沒跟上」的代理，非定論（人早手動確認過但沒動 updated 也會誤報）→ 故 warn_soft、非阻擋。
- **無 cascade ledger（主網 future）**：判過無關卻沒動來源 `updated` 的會每次 doctor 重報（0-issue 麻痺的反面）；量少時手動 bump 來源 updated 消音。ledger 的 `confirmed/pruned ts` 跳過規則屬主網，後上。
- E2 是關係層守衛四檢的第二條；**E4（V.date < node.updated）實測 51 條背書噴 14 條（27%）良性落後 → 判定噪音產生器，不照原 spec 出**；**E3（意圖鏈斷義）精度天生錯（不分是不是那條決策），緩到 [P2] 決策穩定 ID 才做**。
