# rot-eval — 量化 L3 腐化偵測命中率(設計)

> 狀態:CONVERGED(canary-護審計 loop 待跑)｜日期:2026-06-19
> 觸發:2026-06-19 治理日報 gap 1 —— 「第三道腐化偵測本質上會漏掉約一半的隱性失效,而信心門檻是猜的、沒被量過」。
> 外部依據:STALE 評測(arXiv 2605.06527)指最強模型也只認得出約一半「舊資訊已被新事實推翻」的情況。

## 目標(一句話)

拿 LandmarkMember 圖譜自己的 git 史造一組**有標準答案**的「失效題」,重放 L3(`verification-rot-check.py`)現用的判斷,量出它真實的 **recall / precision**,並把 `CONFIDENCE_THRESHOLD = 0.7` 從「拍腦袋」變成「從門檻曲線挑」。

## 邊界 / 非目標(YAGNI)

- **不改 L3 hook 本身**。這是一把尺,不是新的閘。
- **不自動調門檻、不接 CI 擋線、不改 doctor**。先量出來、看清楚再說。
- **先當 prototype 腳本**,不一步到位塞進 `lumos`。證明數字有意義,才升格成 `lumos rot-eval` 子命令 + 週彙整觸發。
- 不追求大 N 統計顯著;這是**方向儀**,不是精密尺(見〈誠實天花板〉)。
- **執行環境限制(R1 §7)**:C 單元真跑 LLM 靠 `claude -p --model sonnet`(Max 訂閱 CLI,無 API key)→ **只能本機跑**,不上 CI。CI 只跑 monkeypatch 模式驗算分邏輯。升格接週彙整觸發時這條限制要正面寫明。

## 形狀:三個獨立單元

每個單元各自一個小檔、各自可獨立測,放 `scripts/rot-eval/`。

### 單元 A — `mine`(挖候選,不貼標)

掃指定 vault 的 `git log`,抓三類訊號當*候選*失效:

1. **`status: pass→stale`**:某 Verification 節點的 frontmatter `status` 從 `pass`(罕見 `pending`)轉成 `stale` 的 commit。**(R1 刪「或 done」,R2 §F3 修正理由:`status: done` 其實**也**出現在 ~20 個 Verification 節點(意義=E2E lab 已驗證);排除它不是因為「它是 Projects 值」(那是錯的),而是因為 `pass→done` 是**成功完成**、不是腐化轉移。)**
2. **`superseded` / 因推翻進 Archive**:Verification 被標 `superseded`(或補 `superseded_by` 欄位),或被移入 `Verification/Archive/` 的 commit。**(R1 修正:必須排除「按年齡批次歸檔」——實際 vault 的歸檔 commit 常一次 rename 14–34 檔、status 不變,那是壽命到期不是被事實推翻。篩除規則:同一 commit rename ≥5 檔且 status 無變動 → 判定批次歸檔、不收為候選;**(R2 §F7)** 同一 commit rename ≥10 檔者,**不論** status 有無變動一律當批次歸檔、標記交人複核——擋掉「30 檔裡夾一個 typo status 改動」蒙混成候選。)**
3. **結論被改寫**:節點的 `summary` 區塊或 `KEY:` 行被實質改動的 commit。**(R2 §F4 修正:原列 `decisions[]` 是 Systems 節點欄位,活躍 Verification 節點沒有它,移除。)**

> **訊號完整性(R1 修正,語氣收斂)**:以上是 prototype 覆蓋的**三類最顯性的 git 訊號**(2026-06-19 user 確認沒有漏掉的第四類 *git* 訊號)。另有一類更結構化、非 git 的訊號——每個 Verification 的 `revalidate_when` 欄位明寫「什麼情況需重驗」(如「等級門檻變更」),可配 code diff 比對——**留待升格時評估**,不在 prototype。不宣稱「竭盡」。

每個候選輸出一筆:
```json
{
  "node_path": "Verification/2026-05-xx_foo.md",
  "overturn_commit": "<SHA>",
  "diff_text": "<觸發失效的【那個 code commit】的【完整 diff】(git diff parent..commit,不過濾);見下方來源邏輯。挖不到則 null>",
  "code_commit": "<diff_text 來自哪個 commit SHA(可能 ≠ overturn_commit)>",
  "old_node_text": "<節點在 code_commit 「父」commit 的【全文】(frontmatter+summary+內文),不是只擷結論>"
}
```
寫成 `rot-eval-candidates.jsonl`。**A 只產候選,絕不貼標籤**——標籤是 B 的事。

> **R1 修正(§5,blocker 級語義錯):** `old_node_text` 必須是**整篇節點全文**,不是只取結論。理由:`build_prompt` 的 `verification_text` 線上吃的是節點全文,裡面的 `valid_under` / `revalidate_when` 正是 L3 判「被驗證的符號/契約有沒有被動到」的核心料。只餵結論會讓 LLM 少看關鍵欄位,量出的 recall **系統性偏低**。**(R2 §F6 注:`build_prompt` 會把 `verification_text` 截到 `MAX_VERIFICATION_CHARS_PROMPT=1500` 字;`valid_under`/`revalidate_when` 在 frontmatter 開頭、截斷後仍在,但超長節點的內文會被切——這是線上真實行為,fixture 存全文、交給 `build_prompt` 自己截即可,不另外處理。)**

> **`diff_text` 來源邏輯(R2 §F1+§F2 重寫,推翻 R1 的「code-only diff」框架):**
> 關鍵事實(對 hook 源碼):① L3 的 `get_diff_text()` 跑 `git diff HEAD~1..HEAD` **完全不過濾檔案**,餵給 `build_prompt` 的是**整個 commit 的完整 diff**(code+md+yaml 全包)。code 副檔名過濾只用在 hook 的**早退守衛**(`if not code_files: return 0`)與候選搜尋,**不是**餵進 prompt 的那份。② **因為早退守衛,L3 線上只會在「動到 code 檔的 commit」上跑**——純圖譜 commit(如某些 status→stale 標記)L3 根本不會觸發。
> 所以挖 `diff_text` 必須是「**一個動了 code 的 commit 的完整 diff**」,依序:
> 1. **`overturn_commit` 自身有動 code 檔**(如 `bf29f51` 同 commit status→stale + 改 `MemberLevelService.cs`)→ `diff_text = git diff parent..overturn_commit`(**完整**,不過濾),`code_commit = overturn_commit`。
> 2. **純圖譜 commit**(status→stale / 進 Archive 但無 code 變動)→ 那個圖譜 commit 不是 L3 會遇到的場景,真正該量的是「**造成失效的那個 code commit**」:**時間窗回溯**——**範圍明確化(R3 MAJOR-2)**:取 `overturn_commit` 之前 **30 天內、或最近 10 個 code commit**(先到者為界),找動到該節點 **`valid_under`**(R2 §F2:Verification 記符號的欄位是 `valid_under`,**不是** Systems 才有的 `verified_by`)所列符號的 commit → `diff_text = 那個 code commit 的完整 diff`,`code_commit = 它`。
> 3. **`valid_under` 非結構化時(R3 MAJOR-2)**:部分節點 `valid_under` 是 block scalar 散文(非符號 list),抽不出可比對的符號 → 直接 `diff_text = null`、`drop`,不硬解析。
> 4. 找不到對應 code commit → `diff_text = null`,B 階段標 `drop`(誠實不硬湊;**絕不拿圖譜自身 before/after 假裝成 diff**)。

### 單元 B — `label`(人工確認,產黃金集)

把 `rot-eval-candidates.jsonl` 攤成可讀清單,人(user/Claude)逐筆判:
- `positive` —— 真失效(舊結論確實被新事實推翻)
- `negative` —— code 有動但該節點其實沒失效(量誤報率用)
- `drop` —— 雜訊 / 標不清楚,丟掉

確認過的存 `rot-eval-fixtures.jsonl`(**進 git,是黃金集**)。schema = 候選(含 `old_node_text` 全文 + `diff_text`)+ `"label": "positive|negative"`(drop 不寫入;`diff_text=null` 者一律 drop)。

> 為什麼要 negative:只看 recall 會自我感覺良好。要同時量 precision,才知道調低門檻換來的多抓是真抓還是亂報。

### 單元 C — `run`(重放 + 算分)

對每個 fixture:
0. **複製 L3 的 pre-LLM 閘(R3 BLOCKER-2,保真關鍵)**:L3 `main()` 在叫 LLM 前有三道早退守衛——`code_files` 非空、`MIN_DIFF_LINES(10) ≤ diff_lines ≤ MAX_DIFF_LINES(2000)`。fixture 的 `code_commit` diff **必須先過這三道**;過不了的 → **L3 線上根本不會叫 LLM**,這筆**不餵 LLM**、不計進 LLM 層分母,改記為「**被守衛跳過的 pipeline miss**」(見〈誠實天花板〉層次分離)。不複製這些守衛會把 L3 線上不會碰的 diff 也算進去,recall 虛高。
1. **import 機制(R3 BLOCKER-1)**:hook 檔名 `verification-rot-check.py` 帶連字號,**不能** `import verification-rot-check`(語法錯)。用 `importlib.util.spec_from_file_location("hook", <path>)` + `exec_module()` 載入;monkeypatch 要打 `hook_module.call_claude_sonnet`(模組屬性),不是本地別名。**零重寫**保證跟線上同一把判斷。**型別正確性(R1 §1a)**:`build_prompt(verification_path: Path, ...)` 第一參數是 `pathlib.Path`(hook 取 `.name`),呼叫須 `build_prompt(Path(node_path), old_node_text, diff_text)`。
2. **接回傳(R1 §1b + R3 MINOR-1)**:`call_claude_sonnet(prompt)` 回**完整 dict**(`{invalidates, confidence, reason, affected_symbols, suggested_action}`)、**或 `None`**(失敗路徑),且**不做 schema 驗證**(LLM 格式跑掉時 dict 可能缺鍵)。算分前一律正規化:**`None`、或缺 `invalidates`/`confidence` 鍵 → 視為 `invalidates=false, confidence=0`**(計進分母)。此規則寫死,避免 N 小時偏移。
3. 算(注:`build_prompt` 線上把 `diff_text` 自截到 `MAX_DIFF_CHARS_PROMPT=4000`、`verification_text` 截到 1500;fixture 兩者都**存完整**、交給 `build_prompt` 自截即可,與線上一致——R3 MINOR-2):
   - **recall@0.7** = positives 中,judge 出 `invalidates=true AND confidence>=0.7` 的比率。
   - **precision@0.7** = judge 判為失效的當中,真是 positive 的比率。
   - **門檻敏感性表(R1 §10,降格,不叫「曲線」)**:0.5 / 0.6 / 0.7 / 0.8 各算 recall+precision(positive 數 `< MIN_FIXTURES_FOR_SWEEP`(**= 5**,R3 MAJOR-1 補定義:低於此每格 ≥20%、太粗無意義)時整張表跳過、只報單點 0.7)。**明寫**:N 小(~7 個 positive 時最小粒度 ≈ 14%)下這是**不連續的階梯**,只能判量級差異(「低門檻也多不了幾筆」),**不能拿來精挑最優門檻**——與〈誠實天花板〉一致。
4. 輸出一份 `rot-eval-report.md`(或 json):上述數字 + 每筆 fixture 的 judge 結果(對/錯標出來,方便看它漏哪一半)。

## 資料流

```
vault git 史 ──A.mine──▶ candidates.jsonl ──B.label(人)──▶ fixtures.jsonl(進git)
                                                                  │
                          hook.build_prompt + call_claude_sonnet ◀┘
                                      │
                                  C.run ──▶ report(recall/precision/門檻敏感性表)
```

工具住 `lumos-toolchain`(跟 hook 同源);fixtures 是 vault-specific(LandmarkMember 圖譜史),由 `--vault` 指定。

## 誠實天花板(必須寫進 report 顯眼處)

- **倖存者偏差**:git 史只有「被發現並修掉」的失效;從沒被注意到的那半根本不在樣本裡 → 量出的 recall 是**上限**,真值只會更糟。這把尺證不了「L3 沒漏的那些」。
- **標註也是判斷**:hybrid 的人工確認把「何謂真失效」這個 regress 收斂到人,但沒消滅它。
- **N 小**:fixtures 預期 ~15–30 筆,百分比吵雜。當方向儀(「就算 0.5 也只 60%」這種量級結論)用,別當小數點後的精密值。
- **只量 LLM 判斷層,不量搜尋層(R3 MAJOR-3,最該誠實的一條)**:L3 兩段都會漏——① `find_candidate_verifications`(obsidian 搜尋)根本沒把對的 Verification 撈出來;② `call_claude_sonnet` 撈出來了卻判錯。本 eval **直接把對的 Verification 餵進去**,只量到 ②。所以報告數字是「**LLM 判斷層的 recall**」,**不是整條 pipeline 的 recall**;真實 pipeline 只會**更低**(還要再乘搜尋層的命中率)。標題與結論都不得簡稱「L3 recall」而不加這個限定。
- **守衛跳過 ≠ LLM 漏(R3 BLOCKER-2 衍生)**:被 `MIN/MAX_DIFF_LINES` 守衛跳過的 positive,是 pipeline 層的 miss(設計性跳過),分開記、不混進 LLM 層分母。報告兩個都列:LLM 層 recall + 有幾筆 positive 在進 LLM 前就被守衛/搜尋層擋掉。

## 測試策略

- **`mine`**:給一個合成的 git fixture repo(幾個 commit 含三類訊號各一),驗它抓對候選、欄位齊全、不把無關 commit 撈進來。
- **`run`**:monkeypatch `call_claude_sonnet` 回造假的 `{invalidates,confidence}`,驗 recall/precision/門檻掃描的算分邏輯正確——**不靠真的燒 token**。
- **`label`**:純人工互動,不寫自動測試(它的正確性=人的判斷)。

## 升格條件(prototype → `lumos rot-eval`)

prototype 三單元跑通 + 對 ~15 組真實 fixtures 產出有意義的數字(能講出「L3 在 0.7 大概抓 X 成、漏的多是哪一類」)後,才:
- 收斂成 `lumos rot-eval {mine,run}` 子命令(`label` 維持半互動)。**命名慣例 OK**:lumos 已有二層命令(`guard bind`/`canary record`/`loop status`),`rot-eval mine/run` 與之一致(R1 §9 質疑慣例不符——已剝除:該 finding 漏看了 guard/canary/loop 三組既有二層命令)。
- 接週彙整觸發,把 recall 變成可追蹤的率(對齊「eval 可靠度當成可追蹤的率」——recall 掉了就是 L3 在退化)。
